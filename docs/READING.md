# Required Reading for Flowform Contributors

These books teach the principles we practice. Read them to understand why we do what we do.

---

## Essential (Start Here)

### 1. Clean Code
**by Robert C. Martin**

*The* book on writing maintainable code.

**Key chapters:**
- Chapter 2: Meaningful Names
- Chapter 3: Functions
- Chapter 6: Objects and Data Structures
- Chapter 7: Error Handling

**Why:** Teaches how to write code that's easy to read and change.

[Buy on Amazon](https://www.amazon.com/Clean-Code-Handbook-Software-Craftsmanship/dp/0132350882)

### 2. The Pragmatic Programmer
**by David Thomas & Andrew Hunt**

Timeless wisdom for software craftspeople.

**Key concepts:**
- DRY (Don't Repeat Yourself)
- Broken Windows Theory
- Tracer Bullets
- Code That's Easy to Delete

**Why:** Teaches you to think like a professional.

[Buy on Pragmatic Bookshelf](https://pragprog.com/titles/tpp20/)

### 3. Test Driven Development: By Example
**by Kent Beck**

Learn TDD from the person who invented it.

**Why:** We practice TDD religiously. This shows you why it works.

[Buy on Amazon](https://www.amazon.com/Test-Driven-Development-Kent-Beck/dp/0321146530)

## Architecture (For Core Contributors)

### 4. Clean Architecture
**by Robert C. Martin**

Our codebase follows these patterns exactly.

**Key concepts:**
- The Dependency Rule
- Use Cases
- Interface Adapters
- SOLID Principles in practice

**Why:** This is how Flowform is structured.

[Buy on Amazon](https://www.amazon.com/Clean-Architecture-Craftsmans-Software-Structure/dp/0134494164)

### 5. Domain-Driven Design
**by Eric Evans**

How we model business logic.

**Key concepts:**
- Ubiquitous Language
- Entities vs Value Objects
- Aggregates
- Bounded Contexts

**Why:** Our domain layer uses DDD patterns.

[Buy on Amazon](https://www.amazon.com/Domain-Driven-Design-Tackling-Complexity-Software/dp/0321125215)

---

## Operations (For Infrastructure Work)

### 6. Release It! (2nd Edition)
**by Michael Nygard**

Building production-ready systems.

**Key concepts:**
- Stability Patterns
- Circuit Breakers
- Timeouts and Retries
- Health Checks

**Why:** Production isn't dev. Learn the difference.

[Buy on Pragmatic Bookshelf](https://pragprog.com/titles/mnee2/)

## After Reading

Apply what you learned:
- Write tests first (TDD)
- Keep functions small (Clean Code)
- Name things clearly (Clean Code)
- Follow SOLID (Clean Architecture)

**Reading without practice is useless. Practice is how you learn.**
